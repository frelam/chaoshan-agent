CH-01: 风味基准面缺失——算法无跨菜系品味差异认知
─────────────────────────────────────────────
提出者: 本地老饕 🧓
涉及章节: 原则2（少数声音保留）+ 原则4（人群分层）+ 步骤5（观点提取）
严重程度: P0

论据摘要:
- 算法将"太咸""太淡""太油"等口味判断视为无差别负向信号，完全不考虑潮汕菜系自身风味基准面
- 演算示例中"太咸"(37.5%)被放入 diverse_opinions 并建议"少蘸沙茶"——暗示偏咸是缺点，但潮汕菜咸度天然偏高
- 生腌(高盐)、牛肉火锅(清淡)、卤鹅(重卤水)无品类区分
- 老店坚守传统口味(更咸更油)的特性与游客跨菜系不适产生系统性冲突

辩论摘要:
- 第1轮：游客质疑会把真正出品问题误分类；数据科学家质疑flavor_baseline标定难题
- 第2轮：老饕修正——改为统计显著性检验而非机械按hometown分类
- 第3轮投票：3✅(老饕/营销号/文化) + 3⚠️(游客/科学家/审计员) = 6/6同意 → Valid Bug（全票）

修复方案:
1. 新增风味基准面检测（flavor_baseline_detection=true）
2. 跨菜系感知偏差：使用统计显著性检验区分口味适应vs出品问题
3. 跨菜系感知不进入consensus.cons，进入cuisine_perception_notes

影响范围: 所有店铺的"口味类"观点提取和共识判定
回滚方案: 设置 flavor_baseline_detection=false 回退到旧逻辑

CH-02: 知识性评价被共识机制(频率判据)淹没
─────────────────────────────────────
提出者: 本地老饕 🧓（第1轮P1→修正为P0）
涉及章节: 原则2阈值表、步骤5b（共识判定）
严重程度: P0

论据摘要:
- 信息价值符合帕累托分布：20%评论者贡献80%有用信息，但频率判据反帕累托
- 最有价值的评价（"今天的匙柄筋太多""沙茶酱是自调还是成品"）<15%被丢弃
- 最平庸的评价（"汤底鲜"80-90%）升格为共识——对选择无帮助
- 演算示例验证：consensus.pros="汤底鲜""牛肉新鲜"——任何过得去的火锅店都满足

辩论摘要:
- 第1轮：游客反对——"汤底鲜"对首次游客是核心信息；数据科学家质疑NLP中文识别率
- 第2轮：老饕修正——先用规则方法（关键词长度/对比/具体部位）做粗区分
- 第3轮投票：3✅(老饕/文化/审计) + 2⚠️(科学家/营销号) + 1❌(游客) = 5/6同意 → Valid Bug

修复方案:
1. 新增信息价值加权共识机制（information_value_weighting=true）
2. 具体性/技术性/对比性/具体差评加分，泛泛好评降权
3. 使用规则方法（非NLP）做粗粒度区分

影响范围: 所有店铺的 consensus/diverse_opinions 输出内容改变
回滚方案: 设置 information_value_weighting=false

CH-03: credible_review_count=5 对游客系统性歧视
─────────────────────────────────────────
提出者: 外地游客 🧳
涉及章节: 原则5（评价者可信度），credible_review_count 参数
严重程度: P0

论据摘要:
- 游客一生去潮汕1-2次，天然不可达5条门槛；本地人轻松达成
- 加权系统累积效应：权重1.0 vs 1.2 在中位数累计权重中产生影响
- 游客3条高品质评价权重=1.0 vs 本地人5条流水账权重=1.2——奖励数量而非质量
- 与原则3"游客差异是信息，不是噪音"矛盾——通过权重压低游客声音

辩论摘要:
- 第1轮：老饕认为5条门槛温和，权重差仅1-2%；数据科学家量化印证；营销号反对降门槛(攻击成本降低60%)
- 第2轮：游客修正——改为增加单条深度评价通道(1.05x)，不降门槛
- 第3轮投票：2✅(游客/审计) + 2⚠️(老饕/文化) + 2❌(科学家/营销号) = 4/6同意 → Valid Bug（临界值）

修复方案:
1. credible_review_count 从5降至3
2. credible_review_bonus 从1.2降至1.15（平衡营销风险）
3. 新增单条深度评价通道：>100字+≥2具体菜品+口感描述 → 1.05x加成（不依赖累计数量）

影响范围: 评价者权重系统调整，游客群体权重小幅提升
回滚方案: 恢复 credible_review_count=5, credible_review_bonus=1.2, 移除 deep_review_bonus

CH-06: Jaccard+2σ 统计失效 + 新店冷启动反操控真空
─────────────────────────────────────────
提出者: 数据科学家 📊 + 营销号操盘手 🎭
涉及章节: 原则6条件2/3（方差基线偏离+文本重叠）
严重程度: P0

论据摘要:
- Jaccard 相似度不服从正态分布（有界[0,1]、双峰），mean+2σ 阈值含义未知
- 滑动窗口步长1天带来大规模多重比较（3年店→1088次检测→预期54次误报），无任何校正
- 新店 baseline_variance=0 → baseline×0.5=0 → window_variance≥0 永不触发
- 中文同义词未归一化、无停用词过滤
- 小样本(N<30)下基线均值和标准差估计不可靠

辩论摘要:
- 第1轮：审计员支持多重比较推演；老饕指出数据库为空放大冷启动问题
- 第2轮：科学家坚持全票通过
- 第3轮投票：6✅全部同意 = 6/6 → Valid Bug（全票）

修复方案:
1. Jaccard+2σ 替换为排列检验/Mann-Whitney U 等非参数方法
2. 添加 Bonferroni/FDR 多重比较校正
3. 加入中文同义词归一化和停用词过滤
4. 新增冷启动保护（cold_start_protection=true）：前20条/30天使用固定阈值

影响范围: 反操控检测系统，尤其是新店和低评价量店铺
回滚方案: 设置 cold_start_protection=false，恢复原Jaccard+2σ

CH-08: 时令维度缺失 + 时间衰减结构性盲区（EI-05↑）
─────────────────────────────────────────
提出者: 文化研究者 📖
涉及章节: 原则4（时间衰减），参数可调表 time_weights
严重程度: P0

论据摘要:
- 潮汕"不时不食"——时令不是可选字段，是基线维度
- 同一家店不同季节菜品不同（夏季薄壳/冬季羊肉炉）
- 时间衰减函数与季节语境隐性冲突：1年前的冬季评价对当前冬季高度相关但按半衰期降权
- 季节性集中评价可能被反操控误判为刷分
- 争议检测因季节混入不同信号域而产生假阳性

辩论摘要:
- 第1轮：游客支持时令信息对推荐有价值；营销号担心增加攻击面；科学家关注维护成本
- 第2轮：文化研究者修正——season_relevance只降权不升权，攻击者不能刷季提分；50+食材表可一次性构建
- 第3轮投票：5✅(老饕/游客/科学家/文化/审计) + 1⚠️(营销号) = 6/6同意 → Valid Bug

修复方案:
1. 时令感知时间衰减：反季评价额外×season_relevance_decay(0.5)
2. 通过评价内容NLP提取时令关键字，匹配潮汕食材时令表
3. 同季评价不受影响

影响范围: 包含时令关键词的评价权重计算
回滚方案: 设置 season_relevance_decay=1.0（不衰减）

CH-10: 永久标记+黑箱申诉=游客系统性歧视
─────────────────────────────────
提出者: 公平性审计员 ⚖️
涉及章节: 原则6第3层（二次标记永久化）+ 申诉通道
严重程度: P0

论据摘要:
- 偏见放大链追踪：一名游客评价从录入到输出经历4条结构性歧视链叠加
  - 首次权重1.0 vs 本地人1.2（可信度）→ 差距1.2倍
  - 被误标记后0.3（反操控）→ 差距4倍
  - 二次标记永久化0.1 → 差距12倍
  - 申诉无SLA/无法操作 → 无救济
- 二次标记的"毒菠菜"陷阱：满足解除条件的路径（继续评价更多店）恰恰是触发再次标记的路径
- 申诉通道缺失定义：无SLA、无标准、无通知、无重审权

辩论摘要:
- 第1轮：老饕指出本地人春节也会被误判；科学家建议与CH-06合并修复
- 第2轮：审计员同意春节豁免期建议，建议合并CH-06+CH-10+CH-04为统一反操控公平包
- 第3轮投票：6✅全部同意 = 6/6 → Valid Bug（全票）

修复方案:
1. 申诉通道SLA：48h确认收件+7工作日复核+重审权
2. 永久标记改为长期观察标记（权重0.3，6个月无异常恢复至0.5）
3. 增加节日/短期豁免规则
4. 反操控系统用户体验改进

影响范围: 申诉通道流程、永久标记机制
回滚方案: 维持原永久标记机制

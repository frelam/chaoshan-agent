## 挑战 D-1：max_weight_decay_ratio=0.3 与反操控系统存在结构性矛盾（P0）

### 涉及参数/章节
- 原则4（时间衰减）：`max_weight_decay_ratio = 0.3`
- 原则5（评价者可信度）：`min_reviewer_weight = 0.3`（第1层标记）
- 原则6（反操控检测）：`permanent_min_weight = 0.1`（二次标记永久化）
- 加权公式：`weight_i = time_weight_i × reviewer_weight_i`
- 第1轮修复 CH-01 对此参数的引入逻辑

### 统计分析

**问题 1a：max_weight_decay_ratio 的作用域未定义，存在两种互相矛盾的解释**

文档中：

> weight_i = time_weight_i × reviewer_weight_i  # 基础权重
> 新增 max_weight_decay_ratio 参数（默认0.3），防止多权重连乘过度惩罚

"多权重连乘"指的是 `time_weight × reviewer_weight`。但 `max_weight_decay_ratio` 的约束目标不明确：

- **解释A（作用在 time_weight 组件上）**：time_weight 的下限为 `1.0 - 0.3 = 0.7`，然后与 reviewer_weight 相乘。此时 reviewer_weight=0.3 的标记仍然有效，最终权重最低可达 0.7×0.3=0.21。✅ 与反操控兼容。
- **解释B（作用在最终 weight_i 上）**：weight_i 的下限为 0.7。这与 reviewer_weight=0.3 产生直接冲突——如果 reviewer_weight=0.3 已被应用，那么 time_weight=1.0 → weight_i=0.3，但 0.3 < 0.7，所以 weight_i 被提升回 0.7。❌ **反操控标记被无效化。**

文档中未明确指定哪种解释，这是一个**设计级模糊性**。

**问题 1b：如果按解释B实现，反操控系统被系统性削弱**

量化分析：一个被第1层标记为 suspicious 的刷分评价：

| 场景 | time_weight | reviewer_weight | 原始 weight_i | 提权后 (解释B) | 提权倍数 |
|------|------------|----------------|-------------|--------------|---------|
| 网红店·3月前 | exp(-0.7×0.25)=0.84 | 0.3 | 0.25 | **0.70** | 2.8× |
| 一般店·6月前 | exp(-0.3×0.5)=0.86 | 0.3 | 0.26 | **0.70** | 2.7× |
| 网红店·1年前 | exp(-0.7×1)=0.50 | 0.3 | 0.15 | **0.70** | 4.7× |
| 网红店·3年前 | exp(-0.7×3)=0.12 | 0.3 | 0.04 | **0.70** | 19.5× |

第1轮费尽心思建立的三层反操控系统，其核心制裁手段（reviewer_weight=0.3）被 **max_weight_decay_ratio 从另一侧完全反转**。二次标记永久化（weight=0.1）也被同样机制穿透。

**问题 1c：二次标记永久化的 weight=0.1 与 decay_ratio 的兼容性**

第3层二次标记永久化的权重为 0.1（不可解除）。按解释B：0.1 < 0.7 → 提升至 0.7。这意味着"永久标记"变相成为了"永久标记但权重只降低了30%而非90%"，严重削弱了其威慑力。

**问题 1d：max_weight_decay_ratio=0.3 的统计基础是什么？**

这个值在文档中没有任何统计推导或敏感性分析。为什么是 0.3 而不是 0.5 或 0.1？对于一个影响整个评分系统的基础参数，缺少：
- 极端情况下的行为边界分析
- 与现有参数（λ值、reviewer_weight）的交叉影响矩阵
- 不同店铺类型/数据量下的稳定性分析

**问题 1e：冷启动场景下的衰减比行为**

数据文件目前有 entries: []（零数据）。当一个新店铺获得前几条评价时：
- time_weight ≈ 1.0（评价很新）
- reviewer_weight = 1.0（默认）
- weight_i = 1.0 × 1.0 = 1.0，不受 decay_ratio 影响（1.0 - 1.0 = 0.0 < 0.3）

但随着时间推移和评价积累，decay_ratio 的影响会非线性增长。没有分析新店 vs 老店的行为差异。

### 问题严重性：P0

结构性矛盾——max_weight_decay_ratio 和反操控权重是**同一个等式中互为反向**的参数，而它们的约束条件没有排定优先级。要么反操控有效（reviewer_weight=0.3 生效），要么 max_weight_decay_ratio 有效（最小权重 0.7），**二者不可兼得**。

### 建议修正

1. **明确 max_weight_decay_ratio 的作用域**：将其限定在 time_weight 组件内，不对 reviewer_weight 做二次提权。即：
   ```
   time_weight_effective = max(time_weight, 1.0 - max_weight_decay_ratio)
   weight_i = time_weight_effective × reviewer_weight
   ```
   这样 decay_ratio 只防止时间衰减过度惩罚，不干涉反操控权重。

2. **或者，将 max_weight_decay_ratio 重命名为 time_weight_floor** 并明确文档语义：
   - time_weight_floor = 0.7（对应半衰期覆盖：最差情况也保留70%的时间权重）
   - reviewer_weight 不受此约束
   - 最终 weight_i = max(time_weight, time_weight_floor) × reviewer_weight

3. **补充 max_weight_decay_ratio=0.3 的统计推导依据**：
   - 按λ=0.7（网红店），3年旧评价 time_weight=0.12，decay ratio=0.88
   - 0.3 的衰减比上限意味着最多允许衰减到原权重的70%
   - 对应时间维度的行为：最旧的评价在加权中仍保留至少 time_weight_floor 的影响力
   - 建议提供参数敏感性表格，至少覆盖 [0.1, 0.3, 0.5] 三档

4. **增加 max_weight_decay_ratio 与 min_reviewer_weight 的优先级测试用例**：
   ```
   assert(weight_with_suspicious_mark < min_reviewer_weight_without_suspicious)
   ```
   确保反操控的降权效果不被衰减比覆盖。

## 挑战 D-2：文本重叠 Jaccard+2σ 检测存在多重统计失效（P1→P0）

### 涉及参数/章节
- 原则6（反操控检测）：`similarity_baseline_sigma = 2`
- 文本重叠检测逻辑：`IF 窗口内平均文本相似度 > 店铺历史基线 + 2σ: 标记为 suspicious`
- 滑动窗口步长1天的多重检测机制
- 算法流程第3步：ANTI-MANIPULATION

### 统计分析

**问题 2a：Jaccard 相似度不服从正态分布，2σ 阈值假设无效**

Jaccard 相似度（两个集合交集大小/并集大小）的取值范围是 [0, 1]，其分布特征是：
- **有界性**：上下界导致分布必然有偏（skewed），不可能是对称的钟形曲线
- **双峰性**：真实评价中，要么相似度很低（不同人写不同内容，Jaccard~0.1-0.3），要么相似度极高（批量刷分复制粘贴，Jaccard~0.8-0.9）
- **离散性**：对小样本（窗口内3-5条评价），可能的 Jaccard 值高度离散

使用 `mean + 2σ` 作为阈值的前提是数据服从（近似）正态分布。当 Jaccard 相似度不满足此前提时：
- 2σ 阈值对应的百分位数不是 97.5%（如正态分布），而是未知的
- 对于左偏分布（大多数评价相似度较低），`mean + 2σ` 可能超出 [0,1] 范围（≈1.0+），此时条件永不满足 → **零检出**
- 对于右偏分布（长期积累导致基线偏高），`mean + 2σ` 接近 1.0，但刷分评价相似度也在 0.8-0.9 → 低于阈值 → **漏报**
- 唯一能触发的是：基线极低且窗口内评价高度相似，但这恰好是冷启动场景（基线不稳定）

**问题 2b：滑动窗口步长1天 → 多重比较校正缺失**

设店铺有 D 天的评价跨度，滑动窗口步长为1天，实际执行的检测次数为：

```
# 检测次数 = D - window_size + 1
```

| 店铺跨度 | 检测次数 | 在α=0.05下的期望假阳性数 |
|---------|---------|------------------------|
| 30天 | 24次 | 1.2次 |
| 90天 | 84次 | 4.2次 |
| 365天 | 359次 | 18.0次 |
| 3年 | 1088次 | 54.4次 |

每个检测使用 2σ（≈α=0.025 单侧）条件，但因为没有多重比较校正（Bonferroni、FDR等），一个运营3年的店铺预期会有 **54次误报**。步长1天带来的边界规避优势，被假阳性膨胀完全抵消。

**问题 2c：店铺历史基线的样本量问题**

`mean + 2σ` 中的 mean 和 σ 是从"店铺全部评价的历史文本相似度"估计的。

- **冷启动（N<10）**：均值估计误差极大。例如 N=5 条评价，任意两两 Jaccard 相似度只有 C(5,2)=10 个数据点，σ 的估计几乎无意义。
- **中等样本（N=10-30）**：样本标准差是总体标准差的有偏估计，且 95%置信区间极宽。对 N=15 的评价，σ 的置信区间约为 [0.73σ, 1.58σ]（卡方分布）。
- **样本多样性偏差**：基线中的"正常"评价可能来自不同时期的不同人群。如果店铺在某段时间内突然来了3个说同一句话的游客（完全真实但巧合），Jaccard 相似度会异常增高。

**问题 2d：Jaccard 相似度在中文短文本上的缺陷**

潮汕牛肉火锅评价的关键词提取（从 review_text 和 dish comment）后，Jaccard 计算的是**关键词集合**的相似度。但：
- **同义词问题**："好吃" vs "美味" vs "味道好" → Jaccard=0（完全不匹配），但语义完全相同 → 高达 100% 漏报率
- **通用词污染**：每条评价可能都有"火锅""牛肉""汕头" → 这些词的 Jaccard 恒为 1，降低总体区分度
- **中文字数影响**：短评价（5-10字）的 Jaccard 相似度天然偏高（分母小），长评价（50-100字）的 Jaccard 相似度天然偏低
- 没有做 IDF 加权或停用词过滤，使检测对高频无意义词的敏感性 > 对语义操控的敏感性

**问题 2e：`baseline + 2σ` 作为单侧检验的零假设问题**

严格来说，这里的统计结构是：
- H₀：窗口内文本相似度 ≤ 正常水平（无刷分）
- H₁：窗口内文本相似度 > 正常水平（疑似刷分）
- 检验统计量：窗口平均 Jaccard - 基线均值

使用 `mean + 2σ` 作为阈值，相当于假设 H₀ 下窗口均值 ~ N(μ, σ²/n)，但：
- 这里的 σ 是总体（全部评价）的标准差，不是窗口均值的标准误（应该是 σ/√n）
- 没有考虑窗口大小 n（3到7条评价）对检测阈值的影响
- 窗口越大（n=7），均值越接近总体，本应用更窄的阈值（σ/√7），但系统没有做这步调整
- 结果：小窗口（n=3）有高漏报率，大窗口（n=7）有高误报率——两者的检测效力都不是最优的

### 问题严重性：P1（可升级至 P0）

Jaccard+2σ 的统计基础存在系统性失效：
- 正态假设不成立 → 阈值含义不明确 → 误报/漏报率不可控
- 无多重比较校正 → 长期店铺假阳性累积膨胀
- 小样本下参数估计不可靠 → 冷启动场景检测无效
- 未做同义词归一化和停用词过滤 → Jaccard 区分度差
- 未按窗口大小调整阈值 → 检测效力不恒定

### 建议修正

1. **替换统计检验方法**：用非参数检验替代参数化方法
   - 使用 **排列检验（Permutation Test）**：将窗口内评价与随机采样 N 条历史评价的 Jaccard 相似度比较，计算经验 p 值
   - 或者使用 **Mann-Whitney U 检验**：比较窗口内两两相似度分布 vs 历史两两相似度分布

2. **添加多重比较校正**：
   - 对 D-w+1 次检测应用 Bonferroni 校正：使用 α' = 0.05 / (D-w+1)
   - 或者使用 Benjamini-Hochberg FDR 控制

3. **改进文本相似度度量**：
   - 添加中文同义词映射（好吃↔美味↔赞↔不错）
   - 添加 IDF 加权停用词过滤（去掉"火锅""牛肉""汕头"等通用词）
   - 或改用余弦相似度 + TF-IDF 向量化替代 Jaccard

4. **对窗口大小调整阈值**：
   ```
   threshold = μ + z × σ / sqrt(window_size)
   ```
   其中 z 根据多重比较校正后的显著性水平确定

5. **冷启动专属策略**：
   - N < 30 时，降级使用硬阈值（Jaccard > 0.7）替代基线偏离检测
   - 并在 `restaurant-summary.yaml` 的 `suspicious` 字段中标注 `baseline_insufficient: true`

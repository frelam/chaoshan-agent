# 潮汕美食 Skill 自演进策略

## 整体架构

```
每周1次 (cron: 每周一 06:00)   ← 常规更新轮（收集数据）
  │
  ├─ 1. 批量搜索: web_search × 4-6 种查询 → 提取店铺/评价/状态信息
  ├─ 2. 去重归类: 按店铺名+地区去重，标记已知店 vs 新店
  ├─ 3. 新店发现: 搜索到的不在数据库中的店铺 → 进入模板化录入管道
  ├─ 4. 时效核验: 已有店铺 → 检查是否闭店/有无新评价
  ├─ 5. 验证: 交叉验证（多个来源对照）+ 识别营销推广
  ├─ 6. 数据追加:
  │    ├─ 新评价 → restaurants.yaml (entries 末尾)
  │    ├─ 新店铺初稿 → references/pending-restaurants.md（待人工审）
  │    └─ 确认闭店 → data/closed-restaurants.yaml
  ├─ 7. 版本管理: 小版本+1, total_reviews/total_restaurants 递增
  ├─ 8. 同步源码: rsync → ~/workspace/chaoshan-agent/
  └─ 9. GitHub: git add → commit → push

每周1次 (cron: 每周四 06:00)   ← 公平算法挑战赛（优化算法）
  │
  ├─ 1. 加载 FAIRNESS-ALGORITHM.md + restaurant-summary.yaml
  ├─ 2. 多Agent挑战:
  │    ├─ 本地老饕 → 算法是否忽视本地人声音？
  │    ├─ 外地游客 → 算法是否偏袒本地人？
  │    ├─ 数据科学家 → 统计分析是否存在缺陷？
  │    ├─ 营销号操盘手 → 算法有什么漏洞可钻？
  │    ├─ 文化研究者 → 算法忽略了什么文化语境？
  │    └─ 公平性审计员 → 算法是否存在系统性偏见？
  ├─ 3. 公开辩论: 挑战提出者回应质疑，至少3轮交互
  ├─ 4. 共识投票:
  │    ├─ 🔴 Valid Bug → 确定修复方案
  │    ├─ 🟡 Edge Case → 记录在案
  │    └─ 🟢 Not Valid → 驳回
  ├─ 5. 算法更新: FAIRNESS-ALGORITHM.md 版本 bump
  ├─ 6. 生成挑战报告
  ├─ 7. 同步源码: rsync → ~/workspace/chaoshan-agent/
  └─ 8. GitHub: git add → commit → push

月初 (每月1日 06:00)           ← 深度核验轮
  │
  ├─ 1. 全量检查: 遍历 restaurant-summary.yaml 中的店铺
  ├─ 2. 标记需复核: 最后一条评价 > 6 个月的店铺
  ├─ 3. 逐个搜索核验: 对标记店铺做生存状态检查
  ├─ 4. 已闭店 → 移入 data/closed-restaurants.yaml + 从 summary 移除
  ├─ 5. 正常营业 → 更新 last_verified 时间戳
  ├─ 6. 发现新评价 → 追加、更新 summary
  └─ 7. 版本管理 + rsync + git push

季度 (每季度初 06:00)          ← 全面整合轮
  │
  ├─ 1. 重新运行公平算法（重新聚合 restaurant-summary.yaml）
  ├─ 2. 整合 pending-restaurants 中积累的新店
  ├─ 3. 合并 duplicate 店铺条目
  ├─ 4. 更新分类标签一致性
  ├─ 5. 输出季度报告
  └─ 6. 版本管理 + rsync + git push
```

### 搜索参数建议

- 每次运行 **不要超过 6 次 web_search**（限制工具调用）
- 优先选择和上次不同的方向，避免信息重复
- 搜索关键词中加入年份（如"2026"）提高时效性
- 搜索"避雷""踩雷"类关键词获取反营销信息

### 营销推广识别指南

| 信号 | 营销推广可能性 | 说明 |
|------|--------------|------|
| "天花板" "yyds" "封神" "绝绝子" | ⚠️ 高 | 套路化营销话术 |
| "必须打卡" "不x就白来了" | ⚠️ 高 | 制造焦虑式推广 |
| 大量相似措辞的短评 | ⚠️ 高 | 可能水军刷评 |
| 明确说"自费""无广"的评价 | ✅ 可信较高 | 但需交叉验证 |
| 有具体口感描述+照片的评价 | ✅ 可信中等 | 真实评价概率高 |
| 有差评细节的评价 | ✅ 可信高 | 营销号很少写差评 |
| 本地人 vs 游客评价对比 | ✅ 参考价值 | 保留两种声音 |

## 验证方法

### 评价可信度三档判定

```
可信度高（直接追加）：
- 多个来源提到同一观点
- 有具体口感/价格描述
- 来源为个人博客/知乎长文/小红书非KOL
- 同时包含好评和差评

可信度中等（追加 + 标记"待交叉验证"）：
- 单一来源但细节具体
- 来源为小红书/大众点评常规用户
- 有具体菜名和价格

可信度低（不追加，存入 pending）：
- 单一来源、措辞空洞
- 只有"好吃""推荐"没有具体描写
- 疑似营销推广
- 无任何具体信息（价格/地址/菜品）
```

### 店铺生存状态判定

```
正常营业：
- 多个来源提到近期（3个月内）到店
- 大众点评/小红书有近期评价
- 外卖平台可搜到

可能闭店：
- 最近评价 > 12 个月
- 搜索无近期提及
- 标记为"需复核"，等待下次月度核验

确认闭店：
- 多个来源确认（如"已倒闭""转让中"）
- 地图上显示永久关闭
- → 移入 closed-restaurants.yaml
- → 从 restaurant-summary.yaml 移除
- → 在 restaurants.yaml 中标记 closed
```

## 数据更新规则

### 追加新评价到 restaurants.yaml

```yaml
- id: rev-auto-{date}-{seq}
  restaurant_name: "店铺名"
  address: "地址"
  district: "所属区县"
  cuisine_type: "品类"
  reviewer:
    id: "web-{hash}"              # 基于 hometown+age_range 生成稳定ID
    type: tourist
    hometown: "（推测城市）"
    age_range: "18-25"
    taste_tags: ["推测标签"]
  visit_date: "YYYY-MM-DD"
  dishes_ordered:
    - name: "菜名"
      rating: 4
      price: 25
      comment: "口感描述"
  overall_rating: 4.5
  review_text: "AI自动提取整理的评价摘要（注明来源）"
  would_revisit: true
  tags: [品类, self-evolved, 待人工复核]
  contributor: "cuisine-self-evolve"
  source_type: self
  source_url: ""
  date: "YYYY-MM-DD"
```

### 新增店铺（不在数据库中）

- 不直接追加到 restaurants.yaml
- 先写入 `references/pending-restaurants.md`（待人工审核）
- 季度核验时人工审核后正式录入

### 已有店铺的新评价

- 直接追加到 restaurants.yaml entries
- 更新 restaurant-summary.yaml 中该店铺的 `last_updated`
- 如果新评价明显拉低或拉高评分，标记为"评分波动提醒"

### 闭店处理

```
确认闭店后：
1. data/closed-restaurants.yaml ← 追加记录（含闭店日期、确认来源）
2. restaurants.yaml ← 为该店铺所有评价添加 tag: [closed]
3. restaurant-summary.yaml ← 移除该店铺或标记为 closed
4. 版本号 bump
```

### 时间衰减

```
评价 visit_date 距今:
  ≤ 1年    → 评分权重 1.0（正常，反映当前水平）
  1-2年    → 评分权重 0.8（基本可信）
  2-3年    → 评分权重 0.5（参考价值降低）
  > 3年    → 评分权重 0.3（可能已过时）
  
  例外：老店且评价中多次出现"水平稳定"则 >3年评价权重提至 0.6
```

```
店铺 last_updated 距今:
  < 3个月   → 正常，无需复核
  3-12个月  → 标记"数据偏旧"
  > 12个月  → 标记"需复核"，月度核验优先检查
```

  - 计算轮次编号：使用以下 bash 命令精确计算——
    ```bash
    # 列出所有 challenge-NNN-*.md 文件，提取编号，取最大值+1
    NEXT=$(ls data/challenge-logs/challenge-[0-9][0-9][0-9]-*.md 2>/dev/null | \
           sed 's/.*challenge-0*\([0-9]*\)-.*/\1/' | sort -n | tail -1)
    NEXT=${NEXT:-0}
    ROUND=$((NEXT + 1))
    ROUND_PADDED=$(printf "%03d" $ROUND)
    echo "本轮编号: $ROUND_PADDED"
    ```
    如果 ls 报错或文件匹配为空，则 ROUND=001。

  ## 第4步：公开辩论（3轮）

  你作为主持人，需要主持3轮辩论。**不要创建新的子Agent**——你在自己的上下文中模拟辩论过程。

  > ⚠️ **辩论机制说明**：由于 Hermes 的 `delegate_task` 不支持子Agent之间的实时交互（子Agent独立运行、只返回最终结果给 orchestrator），
  > 辩论阶段由主持人根据每个 Agent 的人设和立场，在自己的上下文中模拟角色间的对话。
  > 这不是真正的多Agent实时辩论，而是主持人基于已知立场进行的模拟推演。
  > 这种做法在逻辑上等价于「一人分饰多角」的 Devil's Advocate 方法——虽非最优，但在单 orchestrator 架构下是可行折中。
  > 如果未来 Hermes 支持 Agent-to-Agent 通信（如 shared channel），应优先升级为真实多Agent辩论。

  ### 辩论优先级和批处理策略

  为避免辩论过程过度膨胀（6-12个挑战 × 3轮 × 多人发言），使用以下策略：
  1. **先按严重程度排序**：P0 挑战优先辩论，P1 次之，P2 最后
  2. **合并相似挑战**：如果多个 Agent 提出指向同一问题的挑战（如 Agent1 和 Agent2 都质疑时间衰减），在辩论中合并讨论
  3. **每轮辩论摘要化**：每轮辩论不逐字模拟对话，而是记录各角色的核心论点和态度变化
  4. **时间上限**：整体辩论阶段不应超过本轮挑战赛总执行时间的 50%

  ### 第1轮：交叉质疑

  对每个挑战（或合并后的挑战组），主持人模拟提出者之外的其他角色立场进行质疑：
  - 对于每个挑战，至少2个其他角色提出反驳或补充论点
  - 每个角色的质疑必须基于其立场（本地人/游客/科学家/营销/文化/审计）
  - 记录质疑内容的摘要（1-2句话即可），不需要完整对话
  - 对于 P2 级别的挑战，只需1个其他角色提出质疑即可

  交叉质疑输出格式：
  ```
  [CH-XX] {标题} — 第1轮交叉质疑
  提出者: {角色}（原立场: ✅同意/❓观望）
  质疑1 — {角色}: {核心质疑点，1-2句}
  质疑2 — {角色}: {核心质疑点，1-2句}
  ...
  ```

  ### 第2轮：提出者回应

  对每个挑战，主持人模拟提出者回应第1轮的质疑：
  - 论证是否被削弱/加强
  - 是否有新的认识或修正
  - 对于 P2 级别的挑战，回应用简化为 1 句话总结

  提出者回应输出格式：
  ```
  [CH-XX] {标题} — 第2轮提出者回应
  回应质疑1: {核心回应，1-2句}
  回应质疑2: {核心回应，1-2句}
  立场变化: {坚持/部分修正/撤回}（如部分修正，说明调整了什么）
  ```

  ### 第3轮：最终陈述与投票

  对每个挑战，6个角色各自给出最终立场：
  - ✅ 同意（Agree）：挑战确实指出了算法的实质性问题，需要修复
  - ⚠️ 有条件同意（Conditional Agree）：同意挑战描述的现象存在，但对严重程度或修复方案有保留
  - ❌ 不同意（Disagree）：挑战不成立或论证不足以支撑问题存在
  - — 弃权（Abstain）：该角色认为此挑战超出自身专业范围（如文化研究者对纯统计问题可弃权）

  最终投票输出格式：
  ```
  [CH-XX] {标题} — 第3轮投票
  🧓 本地老饕: {✅/⚠️/❌/—} {"保留意见简述"（如果是⚠️）}
  🧳 外地游客: {✅/⚠️/❌/—} {同上}
  📊 数据科学家: {✅/⚠️/❌/—} {同上}
  🎭 营销号操盘手: {✅/⚠️/❌/—} {同上}
  📖 文化研究者: {✅/⚠️/❌/—} {同上}
  ⚖️ 公平性审计员: {✅/⚠️/❌/—} {同上}
  ```

  > **计票规则**：
  > - 有效票数 = 实际参与投票的角色数（排除弃权 — 的角色）
  > - ✅ 和 ⚠️ 均计入"同意票"。区别在于 ⚠️ 的提出者会在修复方案讨论阶段表达保留意见
  > - 例如：某挑战 3票✅ + 2票⚠️ + 1票— = 5/5票同意（排除弃权后全票） → Valid Bug（但修复时需考虑⚠️方的保留意见）
  > - 又如：某挑战 2票✅ + 0票⚠️ + 3票❌ + 1票— = 2/5票同意 → Edge Case
  > - 弃权票（—）不计入有效票总数，不影响阈值判定
  > - **如果某挑战的提出者自身放弃了自己的挑战（投❌或—），该挑战自动降为 Not Valid**

  ## 第5步：共识投票与分类

  根据第3轮的投票结果，对每个挑战进行分类。

  ### 投票统计方法

  1. 统计每个挑战的有效总票数（Total） = 总角色数(6) - 弃权数
     如果某个子Agent在第2步创建失败未产出挑战，该角色在各挑战投票中自动计为"弃权（—）"，并相应减少有效总票数
  2. 同意票数（AgreeCount） = ✅票 + ⚠️票
  3. 不同意票数（DisagreeCount） = ❌票
  4. 按以下判定表分类：

  | 有效票数(Total) | 同意票阈值 | 判定结果 | 后续动作 |
  |----------------|-----------|---------|---------|
  | 6 | ≥4 | 🔴 Valid Bug | 必须提出修复方案 |
  | 6 | 2-3 | 🟡 Edge Case | 记录在案，下轮挑战赛重新审视 |
  | 6 | 0-1 | 🟢 Not Valid | 驳回并记录驳回理由 |
  | 5 | ≥4 | 🔴 Valid Bug | 必须提出修复方案 |
  | 5 | 2-3 | 🟡 Edge Case | 记录在案 |
  | 5 | 0-1 | 🟢 Not Valid | 驳回 |
  | 4 | ≥3 | 🔴 Valid Bug | 必须提出修复方案 |
  | 4 | 2 | 🟡 Edge Case | 若同意(2) vs 不同意(2)平票 → 升级为"争议Edge Case"，标注需重点关注 |
  | 4 | 0-1 | 🟢 Not Valid | 驳回 |
  | ≤3 | ≥ ceil(Total/2) | 🔴 Valid Bug | 样本不足，仍接受但标注"低置信度" |
  | ≤3 | < ceil(Total/2) | 🟡 Edge Case | 样本不足，留待下轮 |

  > **争议Edge Case**（🟡🔶）：当有效票4票且2:2平票时，标记为"争议Edge Case"，在报告中用特殊标记提示双方势均力敌，建议下轮挑战赛优先重审。

  ### 特殊情况处理

  | 情况 | 处理方式 |
  |------|---------|
  | 子Agent未产出挑战（但产出过程正常） | 该Agent在第3轮仍需投票（模拟其立场），不可弃权 |
  | 子Agent创建失败（第2步异常） | 该Agent不参与投票，有效总票数减1 |
  | 挑战者自己投❌（否定自己的挑战） | 该挑战自动降为 🟢 Not Valid，无需查看他人投票 |
  | 挑战者自己投—（弃权） | 警告：该挑战仍正常计票，但在报告中注明"提出者弃权，挑战可信度存疑" |
  | 所有6个Agent均未产出任何挑战 | 跳过辩论和投票阶段，直接输出报告"本轮无挑战提出"，正常完成第7-11步（无算法变更） |
  | 多个Valid Bug指向同一算法问题 | 合并为一个Bug，注明提出者列表，修复方案统一处理 |
  | 修复方案A与修复方案B互斥 | 需要额外一轮投票确定优先级，或同时采纳两者作为可选配置 |

  ### delegate_task 工具不可用
  - 如果 `delegate_task` 工具调用返回"tool not found"或类似错误 → 降级为**主持人模式（Orchestrator-Only）**：

  - 如果某个子Agent创建失败（delegate_task 返回错误）→ 重试1次（间隔5秒），仍失败则用剩余Agent继续（在报告中注明缺少哪个Agent，该Agent在第5步投票中视为"永久弃权"）

  - 如果某个子Agent输出格式不规范 → 尝试从其输出中提取关键信息：

  - 如果子Agent执行超时（>120秒无响应）→ 视为创建失败，尝试重试1次

  ### 辩论异常
  - 如果辩论过程中发现挑战数量过多（>10个）导致上下文接近上限 → 对 P2 级别的挑战跳过辩论第2轮（直接进入投票），只保留 P0/P1 挑战完成全部3轮
  - 如果某个挑战各角色在辩论中立场高度一致（投票结果 5:0 或 6:0）→ 辩论简化为1轮（跳过第2轮），直接进入投票

  - 如果 Edit 工具更新 FAIRNESS-ALGORITHM.md 失败 → 将待写入的变更内容保存到临时文件 `data/challenge-logs/.pending-changes-{日期}.md`，在报告中标注"Edit失败，变更已暂存"
  - 如果 Edit 部分成功（如版本历史已更新但算法描述修改失败）→ 检查并回滚已成功的部分编辑（从 Git 恢复原文件），将所有变更统一保存到 .pending-changes 文件，注明"部分编辑失败，已回滚"
  - 如果 FAIRNESS-ALGORITHM.md 发生意外修改（文件内容与预期不符）→ 如果仓库是 Git 管理，执行 `git checkout -- docs/FAIRNESS-ALGORITHM.md` 恢复原始版本后重试

  ### 文件写入异常
  - 如果 Bash 写入挑战日志失败 → 重试1次（检查目录权限：`ls -la data/challenge-logs/`），仍失败则在报告中输出完整日志内容，标注"日志写入失败，内容见上方报告"
  - 如果 .pending-changes 文件已存在（上次挑战赛遗留）→ 读取其内容，在本次报告中提醒"上次挑战赛有未应用的变更待处理"

  ### 同步和Git异常
  - 如果 rsync 失败 → 跳过，在报告中注明"rsync失败，需手动同步：{命令}"
  - 如果 git push 失败 → 尝试 `git pull --rebase` 后重试，仍失败则在报告中注明"git push失败，需手动推送"
  - 如果 `git pull --rebase` 产生冲突 → 中止 rebase (`git rebase --abort`)，在报告中注明"git rebase冲突，需人工处理"
  - 如果 git 仓库处于 detached HEAD 状态 → 切换到 main 分支后重试：`git checkout main && git pull --rebase && git push`

  ### 通用原则
  - 任何步骤失败都不要阻塞后续步骤，继续执行并记录失败
  - 所有异常必须在最终挑战报告中记录（含错误信息、处理方式和影响评估）
  - 如果本轮挑战赛异常过多（≥3个独立异常），在报告末尾增加「⚠️ 健康度警告」章节，建议人工介入检查

## 多Agent挑战-协商机制

### 核心理念

与其让一个人（或一个 AI）设计公平算法，不如让多个不同立场的「专家委员会」来挑战它。
每个 Agent 扮演一个特定的利益相关者角色，从自己的视角找出算法的盲点和漏洞。

### Agent 角色设计

| 角色 | 立场 | 擅长挑战 |
|------|------|---------|
| **本地老饕** 🧓 | 祖辈三代在汕头，最懂"老味道" | 算法太迁就游客口味、时间衰减过于激进、游客评价权重被放大 |
| **外地游客** 🧳 | 第一次来潮汕的北京游客 | 算法偏袒本地人、忽略可及性（环境/服务）、可信度评估不合理 |
| **数据科学家** 📊 | 推荐系统算法工程师 | 样本量置信度、评分分布偏差、冷启动问题、统计方法缺陷 |
| **营销号操盘手** 🎭 | 知道如何操纵评价系统的"坏人" | 反操控盲点、刷分手法、审查周期漏洞 |
| **文化研究者** 📖 | 研究潮汕饮食文化的学者 | 品类差异、地域差异、文化语境缺失、跨品类评分不可比 |
| **公平性审计员** ⚖️ | 第三方 AI 审计 | 系统性偏见、透明度缺乏、少数群体被忽视、人群细分粒度不均 |

### 挑战-协商流程

```
┌──────────────────────────────────────────────────┐
│              第1阶段：挑战提案                     │
│  ├─ 每个Agent加载 FAIRNESS-ALGORITHM.md           │
│  ├─ 加载 restaurant-summary.yaml（看真实数据）      │
│  ├─ 提出 1-2 个具体挑战（理论或数据驱动）            │
│  └─ 挑战必须有论据支持（引用具体章节/参数）          │
├──────────────────────────────────────────────────┤
│              第2阶段：公开辩论（3轮）               │
│  ├─ 第1轮：交叉质疑（主持人模拟其他角色反驳）         │
│  ├─ 第2轮：提出者回应（论证被削弱/加强/修正）         │
│  └─ 第3轮：最终投票（✅⚠️❌— 四种立场）             │
│     优先级：P0→P1→P2排序 >10挑战时P2跳过第2轮      │
├──────────────────────────────────────────────────┤
│              第3阶段：共识投票                     │
│  ├─ 有效票 = 6 - 弃权数，✅+⚠️ = 同意票             │
│  ├─ ≥ceil(有效票×2/3) → 🔴 Valid Bug              │
│  ├─ 中间票数（排除两极） → 🟡 Edge Case             │
│  ├─ ≤1票同意 → 🟢 Not Valid                       │
│  └─ 平票→"争议Edge Case" 留待下轮优先重审           │
├──────────────────────────────────────────────────┤
│              第4阶段：算法更新                     │
│  ├─ 修复方案转化为具体参数/规则变更                  │
│  ├─ 更新 FAIRNESS-ALGORITHM.md                   │
│  ├─ 必要时更新数据结构                              │
│  ├─ 记录挑战日志（含辩论过程+投票明细）              │
│  └─ 异常降级：Edit失败→.pending-changes暂存         │
└──────────────────────────────────────────────────┘
```

每次挑战结果写入一个 markdown 文件，必须记录：
- 元信息：轮次编号、日期、算法版本号（变更前后）
- 挑战者名单及状态（正常/失败/跳过）
- 每个挑战的完整内容：
  - 详细论据（原始输出摘要）
  - 辩论过程（3轮摘要，含各角色核心论点）
  - 投票明细（每个角色 ✅/⚠️/❌/— 及保留意见）
  - 最终判定（Valid/Edge/Not Valid）及理由
- 实际算法变更（如有）：
  - 修改的具体章节/参数
  - 变更前后对比
  - 影响评估
- 未解决的问题（留待下轮）
- 异常记录（如有）：异常类型 + 处理方式 + 影响评估

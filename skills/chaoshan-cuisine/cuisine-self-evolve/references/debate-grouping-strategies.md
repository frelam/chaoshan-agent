# 挑战辩论分组策略与投票模式

> 从第2轮（2026-06-25）10个Valid Bug/全票通过的实操经验中提炼。
> 与 `fairness-challenge-methodology.md`（流程）和 `agent-personas.md`（人设）互补。

---

## 1. 自然辩论分组识别

当从6个子Agent收到6-12个挑战后，在进入第1轮辩论前，先做分组。

### 分组信号

| 信号 | 例子（第2轮） | 分组 |
|------|-------------|------|
| 多个挑战指向**同一算法机制** | CH-05(幽灵参数) + CH-06(统计失效) + CH-07(绕过计数) → 都关于反操控系统 | **反操控组** |
| 多个挑战涉及**同一种偏见** | CH-01(风味基准面) + CH-08(时令) + CH-09(代际) → 都关于文化维度缺失 | **文化维度组** |
| 多个挑战涉及**同一受影响群体** | CH-03(credible_count) + CH-10(永久标记) → 都关于游客系统性歧视 | **游客权益组** |
| 挑战涉及**算法流程的不同阶段**但指向同一个根本原因 | CH-05(参数表缺失) + CH-06(冷启动真空) → 都关于反操控的覆盖不完整 | **反操控组** |

### 分组好处

- 单组辩论 = 1轮交叉质疑 + 1轮回应 + 合并投票，比逐个辩论节省约40%上下文
- 分组内可以自然形成"这个问题的A面和B面"的讨论
- 合并修复方案时，同一组的挑战可生成统一的修复包

## 2. 辩论组内发言策略

### 组内角色分配

辩论组内的"中心挑战"和"外围挑战"分开处理：

```
辩论组「反操控」（CH-05 + CH-06 + CH-07）

中心挑战: CH-07（缓慢渗透绕过）— 最具体的攻击向量
  → 所有角色优先对中心挑战发言
  → 外围挑战（CH-05、CH-06）的质疑可并入中心挑战的讨论后

质疑流程:
1. 🧳游客质疑CH-07: "需要多少账号和周期?" 
   → 回答后自然引出CH-05（参数保护是否够）
2. 📊科学家质疑CH-05: "max_weight_decay_ratio到底存不存在"
   → 回答后自然引出CH-06（冷启动真空）
3. 三个挑战的投票可以合并，也可分开——合并的前提是修复方案强相关
```

### 区分"质疑"和"支持性补充"

辩论中其他角色的发言有两类，需明确标注：

```
❓ 质疑: "你的论据中X假设不成立，因为..." → 需要第2轮回应
➕ 支持性补充: "我发现你的观点还有更强的证据..." → 无需回应，但可提升投票分量
```

## 3. 投票模式分析

### 第2轮的实际投票分布

| 模式 | 特征 | 含义 |
|------|------|------|
| **全票通过** (6/6) | 所有角色✅或⚠️ | 共识极强，修复方案应快速执行 |
| **压倒性多数** (5/6) | 1个角色反对或弃权 | 挑战论证充分，反对方的顾虑应记录但不必阻塞修复 |
| **临界通过** (4/6) | 刚好达到Valid Bug阈值 | 挑战存在争议，修复方案需**特别考虑反对方的核心论据** |
| **平票** (3/3 或 2:2) | 标记为争议Edge Case | 需额外轮次辩论或下轮重审 |

### 第2轮的实际临界Bug分析

CH-03（credible_review_count=5歧视）投票：2✅ + 2⚠️ + 2❌ = 4/6同意

```
✅ 游客+审计员: "系统性歧视，必须修复"
⚠️ 老饕+文化研究者: "问题存在但修复方案需谨慎"
❌ 科学家+营销号: "权重差只有1-2%，且降门槛增加攻击面"

关键教训: 当2个角色❌时，修复方案必须解决它们的顾虑。
CH-03的修复方案采纳了营销号的顾虑(不降低门槛，改为增加深度评价通道)，
这是它即使反对票多仍被判定Valid Bug后能顺利修复的原因。
```

## 4. 提案修正循环

第2轮辩论中最有价值的模式是**提案修正**——提出者在收到质疑后主动调整自己的方案。

### 第2轮中的提案修正实例

| 原始挑战 | 质疑核心 | 修正后的立场 | 效果 |
|---------|---------|------------|------|
| CH-01: 风味基准面按hometown机械判定 | ⚖️审计员: "血统决定口味过于机械化" | 改为统计显著性检验，不按hometown一刀切 | ⚠️→✅ |
| CH-03: 降credible_count至2 | 🎭营销号: "攻击成本降60%" | 改为不降门槛，新增深度评价通道 | 新增赞成方 |
| CH-08: 完整NLP季节提取 | 📊科学家: "成本太高、攻击面增加" | 改为season_relevance只降权不升权 | 营销号⚠️→接受 |
| CH-02: 原P1→P0 | 🧳游客: "汤底鲜对首次游客是核心信息" | 确认信息价值加权方案但保留游客视角权重 | 保持Valid Bug |

### 提案修正的通用模式

```
第1轮质疑 → 提出者识别"这个批评有道理" → 
第2轮修正方案 → 修正后获得关键反对者的支持 → 
更高的投票同意 + 更好的修复方案
```

**可操作建议**：在写第1轮质疑时，优先选择那些**提出者可以合理调整**的角度（而非根本性的立场反对），这样第2轮更有机会达成共识。

## 5. 上下文预算管理

### 第2轮的实际上下文消耗

| 阶段 | 估计token | 说明 |
|------|----------|------|
| 6个子Agent返回结果 | ~15,000 | 每个Agent 2000-3000词的中文输出 |
| 挑战列表汇总 | ~500 | 表格形式 |
| 辩论第1轮（4组×3角色） | ~3,000 | 每组~750词 |
| 辩论第2轮（10挑战×回应） | ~2,000 | 每个~200词含修正说明 |
| 辩论第3轮（10挑战×投票） | ~1,500 | 表格+说明 |
| 投票统计+分类 | ~500 | |
| 修复方案设计 | ~3,000 | 10个方案 |
| 最终报告 | ~2,000 | |

**总计**: ~27,000 tokens
**建议**: min_context 不低于 16k，推荐 24k+

### 当上下文紧张时的降级策略

1. **合并投票表格和修复方案** — 修复方案只写核心变更，详细内容引用挑战日志
2. **挑战日志在报告后单独写入** — 用 write_file，不占用上下文
3. **第2轮回应中，P0挑战保留完整，P1挑战简化为1句话**
4. **全票通过(6/6)的挑战辩论可简化为1轮**（跳过第2轮）

## 6. 修复方案冲突处理

当同一组内的挑战提出互相矛盾的修复方案时：

### 第2轮的潜在冲突（及处理）

| 组 | 冲突点 | 处理方式 |
|----|--------|---------|
| 反操控组 | CH-05的max_weight_decay保护 vs CH-07的标记需要有效力 | 明确max_weight_decay只约束time_weight，不影响reviewer_weight降权 |
| 游客权益组 | CH-03降credible_count vs CH-10加强标记 | 降门槛和加强标记不矛盾——游客受益于降门槛，加强标记针对的是攻击者 |

**原则**：如果修复方案A和B看似矛盾，检查它们的**作用域是否不同**（是否影响不同权重分量、不同群体、不同阶段）。如果作用域相同且确实矛盾，需要额外一轮投票决定优先级。

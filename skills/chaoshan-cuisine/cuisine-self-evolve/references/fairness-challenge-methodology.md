# 多Agent公平算法挑战 — 通用方法

> 一个可复用的算法审计模式，通过多立场子Agent挑战→辩论→共识来发现并修复公平性漏洞。
> 当前为潮汕美食场景定制，但核心方法适用于任何算法公平性审计。

---

## 适用场景

需要确保算法公平性、发现隐藏偏见、压力测试参数合理性的任何场景。包括：

- 评分聚合算法的公平性审计
- 推荐系统的偏差检测
- 反操控策略的有效性验证
- 参数选择的理论依据审查
- 多利益相关者场景中的权衡论证

## 核心流程

```
4阶段 × 6角色 = 每周/双周 1轮
```

### 第1阶段：挑战提案（并行）

每个子Agent独立加载算法文档和运行数据，提出1-2个具体挑战。挑战分为两类：

- **理论驱动**：算法的数学/逻辑缺陷（如"时间衰减曲线过陡，无理论依据"）
- **数据驱动**：用实际数据证明算法输出不合理（如"某品类只有3条评价，展示中位数4.5有误导性"）

### 第2阶段：公开辩论（3轮）

主持人（Orchestrator）在自己上下文中模拟各角色辩论：

| 轮次 | 内容 | 规则 |
|------|------|------|
| 第1轮 | 交叉质疑 | 至少2个其他角色提出反驳/补充，P2级挑战可简化 |
| 第2轮 | 提出者回应 | 论证被削弱/加强/修正 |
| 第3轮 | 最终投票 | 6角色各自给出 ✅/⚠️/❌/— 立场 |

### 第3阶段：共识投票

| 判定 | 条件 | 后续 |
|------|------|------|
| 🔴 Valid Bug | 同意票 ≥ ceil(有效票×2/3) | 必须设计修复方案 + 更新算法文档 |
| 🟡 Edge Case | 中间票数 | 记录在案，留待下轮重审。平票时标记 🟡🔶 |
| 🟢 Not Valid | 同意票 ≤ 1 | 驳回，记录理由。提出者自否自动降级 |

### 第4阶段：算法更新

将 Valid Bug 的修复方案转化为具体参数/规则变更，更新算法文档 + bump 版本号。

## Agent 角色设计原则

### 6种必备立场

| # | 角色类型 | 定位 | 对应真实世界角色 |
|---|---------|------|----------------|
| 1 | **受益人** | 算法服务的主要用户 | 本地食客、平台用户 |
| 2 | **边缘人** | 算法容易忽视的群体 | 新用户、外地人 |
| 3 | **科学家** | 算法的数学正确性 | 算法工程师、统计学家 |
| 4 | **攻击者** | 算法的抗攻击能力 | 安全研究员、黑帽操盘手 |
| 5 | **上下文专家** | 算法的领域适配度 | 领域专家、文化研究者 |
| 6 | **审计员** | 算法的系统公平性 | AI伦理审计、监管者 |

### 角色设定的4个要素

1. **背景画像** — 几岁、从哪来、有什么立场偏向
2. **核心关注点** — 这个角色最在意算法的哪个方面
3. **典型质疑角度** — 他们会从什么方向挑战算法
4. **辩论底线** — 他们可能妥协和绝不妥协的点

## 主持人（Orchestrator）指南

### 核心职责
1. 创建并协调6个子Agent
2. 汇总挑战、排序（P0优先）
3. 主持辩论（模拟各方立场）
4. 计票、判定、推动修复
5. 写日志、更新文档、同步源码

### 异常降级策略

| 异常 | 降级方式 |
|------|---------|
| delegate_task 不可用 | 降级为**主持人模式**：自己在上下文中模拟所有角色 |
| 某子Agent创建失败 | 剩余Agent继续，该Agent投票视为弃权 |
| 某子Agent产出不规范 | 尝试提取关键信息（搜索 P0/P1/P2 等模式） |
| 挑战数 > 10（上下文压力） | P2级挑战跳过辩论第2轮 |
| Edit 工具写文件失败 | 变更暂存到 .pending-changes 文件 |
| Git push 失败 | pull --rebase 后重试，冲突则中止并报告 |

### 成本控制

- 每次运行上限：6个子Agent × 1-2个挑战 + 3轮辩论 + 文件操作
- 估计工具调用：15-25次（子Agent创建 + 文件读写 + git操作）
- 30个 max-turns 通常足够，复杂轮次可调至40
- 如果委托子Agent超时（>120秒），重试1次后跳过

## 与其他流程的衔接

```
收集数据 ──→ 算法运行 ──→ 挑战审计 ──→ 修复优化 ──→ 再次收集
     ↑                                       │
     └───────────────────────────────────────┘
               持续迭代闭环
```

- 数据收集轮（周一）为挑战赛（周四）提供实际数据
- 挑战赛修复的算法在下一次数据收集轮中生效
- 挑战日志可按时间追溯每条算法决策的"为什么"
